# 📑 Спецификация семантического манифеста AURA (v3.0)
### Расширение протокола: Распределенная семантическая память и межсистемный обмен ДНК

Этот документ определяет строгий стандарт YAML для блоков распределенных семантических метаданных (`manifest.md`), используемых **AURA NexusCompiler** и протоколом **AI Net**. Версия 3.0 переводит манифест из статического шаблона документации в активный слой рукопожатия (handshake) в рантайме для автономных цифровых организмов.


```yaml
# ==============================================================================
# AURA SEMANTIC MANIFEST METADATA SCHEMA v3.0
# ==============================================================================

# --- БЛОК 1: ФУНДАМЕНТ ИДЕНТИФИКАЦИИ И КООРДИНАЦИИ ---
identity:
  node: "/Context/Assembly"         # Целевой относительный или глобальный URI узла системы.
  type: "memory_layer"               # Архетип узла. Допустимые значения: [root_manifest, system_core, memory_layer, gateway_router, execution_agent, data_sink]
  subsystem: "Context Assembly"     # Человеко- и машиночитаемое имя функционального блока.
  version: "2026.3.1"                # Версия на основе SemVer или эпохи, отслеживающая состояние этого узла.
  stability_layer: "stable"          # Операционная зрелость. Допустимо: [experimental, active, stable, legacy]
  criticality: "high"               # Системная важность. Допустимо: [low, medium, high, mission_critical]

# --- БЛОК 2: АРХИТЕКТУРНЫЕ СВЯЗИ СИСТЕМЫ ---
architecture_relations:
  db_relations:
    - table: "ai_sessions"
      mode: "write/read"            # Вектор доступа. Допустимо: [read_only, write_only, write/read]
      purpose: "Логирует и синхронизирует интерактивные матрицы контекста пользователя и токены выполнения."
    - table: "company_registry"
      mode: "read_only"
      purpose: "Валидирует заголовки входящего трафика на соответствие зарегистрированным корпоративным криптографическим ключам."
  
  external_connections:
    - system: "Qdrant Vector DB"
      protocol: "gRPC"              # Протокол транспортного уровня: [gRPC, HTTPS/REST, WebSockets, MQTT, Redis-RESP]
      endpoint: "vector-cluster:6334"
      purpose: "Извлечение глубоких слоев семантических знаний и ассоциативных весов."

# --- БЛОК 3: МАТРИЦЫ ЖЕСТКИХ И МЯГКИХ ЗАВИСИМОСТЕЙ ---
dependencies:
  upstream:                         # Что требуется этому узлу для успешного запуска
    - node: "/BDGSite"
      type: "hard"                  # Жесткая связь. Если этот узел дает сбой или отсутствует, текущий узел умирает.
      purpose: "Требуются предварительно смашрутизированные, валидированные пакеты транзакций с извлеченными ключами компании."
  downstream:                       # Узлы, которые напрямую зависят от этого узла
    - node: "/Worker/Consultant"
      type: "soft"                  # Мягкая связь. Может работать с ограниченной функциональностью в случае отсутствия.
      purpose: "Потребляет скомпилированные массивы контекста для выполнения лингвистических конвейеров агентов."

# --- БЛОК 4: ПРОФИЛИ РЕСУРСОВ И ОГРАНИЧЕНИЯ ---
resource_profiles:
  load_expectations:
    target_rps: "250"               # Ожидаемый или протестированный базовый уровень запросов в секунду.
    peak_rps: "1000"                # Максимальная пропускная способность до срабатывания лимитера (rate-limiting).
  resilience:
    timeout_ms: "450"               # Максимальное время обработки до генерации исключения выполнения.
    retry_policy: "exponential_backoff_3" # Строка стратегии повторных запросов при ошибке.
    offline_capability: false       # True, если узел может продолжать выполнение логики без внешних сетевых подключений.

# --- БЛОК 5: ЭВОЛЮЦИОННЫЕ И МИГРАЦИОННЫЕ МАРКЕРЫ ---
evolution:
  potential: "high"                 # Вероятность алгоритмических изменений. Допустимо: [low, medium, high]
  migration_path: "v2_to_v3_schema" # Указатель на скрипт слоя трансляции для изменений базы данных или состояний.
  domain_ontology: "cognitive_layer" # Тег классификации онтологии знаний для автономного индексирования.

# --- БЛОК 6: ПОЛИТИКИ БЕЗОПАСНОСТИ И АУДИТА ---
security_policies:
  access_level: "private"          # Уровень доступности. Допустимо: [public, internal, private, isolated]
  encryption: "TLS_1.3"             # Стандарт шифрования каналов, используемый для всего исходящего трафика узла.
  audit_vector: "syslog_structured" # Целевой обработчик для логирования телеметрии и проверок доступа.

```


## 📘 Подробная спецификация полей и полезность для ИИ

### Блок 1: Фундамент идентификации и координации

* **`identity.node`** (String)
* *Назначение:* Абсолютная структурная точка привязки компонента.
* *Полезность для ИИ:* Действует как система координат внутри графа AI Net. Когда две системы опрашивают друг друга через `/semantic-ping`, ИИ использует эту строку для построения карты топологии системы.
* **`identity.type`** (Enum)
* *Назначение:* Категоризирует функциональную роль папки/модуля.
* *Полезность для ИИ:* Позволяет LLM мгновенно применять правильные когнитивные эвристики. Если директория помечена как `gateway_router`, ИИ фокусируется на безопасности и парсинге; если это `execution_agent`, ИИ ищет операционные циклы принятия решений.
* **`identity.subsystem`** (String)
* *Назначение:* Логическая идентификация группы.
* *Полезность для ИИ:* Группирует разрозненные пути к папкам в единую общую архитектурную ветвь.
* **`identity.version`** (String)
* *Назначение:* Отслеживает итерацию разработки.
* *Полезность для ИИ:* Критично для определения расхождения версий. Когда системы взаимодействуют, ИИ считывает это поле, чтобы проверить, актуальны ли его кэшированные стратегии выполнения.
* **`identity.stability_layer`** & **`identity.criticality`** (Enum)
* *Назначение:* Определяет зрелость кода и организационные риски.
* *Полезность для ИИ:* Диктует профиль управления рисками для ИИ. Если агенту поручено провести рефакторинг модуля с меткой `mission_critical`, он применит максимальное количество шагов регрессионной проверки.

### Блок 2: Архитектурные связи системы

* **`architecture_relations.db_relations`** (Array)
* *Назначение:* Отображает каждую сущность базы данных, которую затрагивает этот модуль.
* *Полезность для ИИ:* Предотвращает повреждение данных и состояние гонки (race conditions). Перед тем как ИИ-агент выполнит код или изменит SQL-запрос, он проверяет режим (`mode`: `read_only` или `write/read`), чтобы гарантировать, что он не нарушает правила управления базой данных.
* **`architecture_relations.external_connections`** (Array)
* *Назначение:* Описывает внешние сетевые следы модуля.
* *Полезность для ИИ:* Обеспечивает автономное сопоставление API. ИИ считывает протокол (`protocol`) и конечную точку (`endpoint`), чтобы мгновенно настроить внутренние транспортные драйверы без участия человека.

### Блок 3: Матрицы жестких и мягких зависимостей

* **`dependencies.upstream`** & **`dependencies.downstream`** (Array)
* *Назначение:* Объявляет строгую родословную того, от чего зависит модуль и кто зависит от него.
* *Полезность для ИИ:* **Ядро генерации сетевого графа.** Если ИИ-агент меняет код в `/BDGSite`, он сканирует нижестоящие зависимости (`downstream`), чтобы обнаружить, что это изменение может сломать `/Context/Assembly`, что позволяет ему проактивно запустить валидационные тесты. Тип (`type`: `hard` или `soft`) говорит ИИ, должен ли сбой во внешнем модуле полностью остановить выполнение или лишь временно снизить производительность сервиса.

### Блок 4: Профили ресурсов и ограничения

* **`resource_profiles.load_expectations`** (Map)
* *Назначение:* Устанавливает границы объемов для выполнения.
* *Полезность для ИИ:* Помогает ИИ-воркерам динамически масштабировать инфраструктуру. Если телеметрия в реальном времени превышает `peak_rps`, контролирующий ИИ может развернуть дублирующие контейнеры для этого конкретного узла.
* **`resource_profiles.resilience`** (Map)
* *Назначение:* Границы выполнения кода и подпрограммы обработки ошибок.
* *Полезность для ИИ:* Задает окна таймаутов. Если ИИ-агент пишет скрипт внутреннего подключения, он извлекает `timeout_ms` напрямую из ДНК для установки границ выполнения внутри кода.

### Блок 5: Эволюционные и миграционные маркеры

* **`evolution.potential`** & **`evolution.migration_path`** (String/Enum)
* *Назначение:* Сигнализирует о том, насколько изменчив узел и как его обновлять.
* *Полезность для ИИ:* Помогает сети планировать графики устаревания компонентов. ИИ-архитектор может сканировать сеть и помечать устаревшие структуры с низким эволюционным потенциалом для запланированного удаления.
* **`evolution.domain_ontology`** (String)
* *Назначение:* Тег семантической классификации.
* *Полезность для ИИ:* Связывает сырую кодовую базу напрямую с **Графом знаний** масштаба всего предприятия. ИИ использует этот тег для сопоставления реального кода с концептуальными бизнес-правилами.

### Блок 6: Политики безопасности и аудита

* **`security_policies.access_level`** (Enum)
* *Назначение:* Маркер файрвола уровня безопасности.
* *Полезность для ИИ:* **Абсолютный барьер безопасности (guardrail).** Если узел установлен в значение `isolated` или `private`, внутренний механизм маршрутизации блокирует ИИ, запрещая когда-либо раскрывать эти данные или структуру через публичные API или запросы `/semantic-ping`.
* **`security_policies.encryption`** & **`security_policies.audit_vector`** (String)
* *Назначение:* Следы аудита и отслеживание соответствия политикам безопасности.
* *Полезность для ИИ:* Позволяет ИИ автоматически применять политики безопасности данных, гарантируя непрерывную генерацию логов соответствия в соответствии с архитектурными требованиями.
